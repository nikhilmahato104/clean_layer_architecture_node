1. Project Overview & Architecture Philosophy

This project implements Clean Architecture (also called Hexagonal Architecture or Onion Architecture) in a Node.js + Express.js + TypeScript environment — mirroring the same patterns used in enterprise NestJS applications.

The Core Rule: Each layer only depends on the layer below it. Business logic (Use-Cases) never imports from Express or Mongoose — it only talks through interfaces.

Architecture Layer Flow

HTTP Layer
Routes Controllers — Handles req/res only. No business logic. Validates input with DTOs.
↓
Use-Cases
StudentUseCase MarksUseCase — Business rules. Calls IDataServices. No Express/Mongoose imports.
↓
Core/Abstracts
IGenericRepository IDataServices — TypeScript interfaces. Framework-agnostic. The contract layer.
↓
Framework
MongoGenericRepository MongoDataServices — Concrete Mongoose implementation. The only place with DB code.
↓
Database
MongoDB Atlas Mongoose Models — Student collection, Marks collection.

Why this architecture?

🔄
Swap Database Easily
If you want to change from MongoDB to PostgreSQL, only the frameworks/ folder changes. Use-cases and controllers stay 100% unchanged.
🧪
Testable Business Logic
Use-cases have no Mongoose or Express dependencies, so you can test business rules by passing a mock IDataServices object.
📦
Module Independence
Each resource (Student, Marks, Course) has its own isolated folder. Adding or removing one resource doesn't affect others.
🚀
Same as NestJS Pattern
This mirrors NestJS's Service + Repository + Module pattern, so migrating to NestJS later requires minimal refactoring.

3. Core Layer — Abstracts, Entities & DTOs

Location: src/core/ — This is the most important folder. It contains zero Mongoose or Express code. Pure TypeScript interfaces that define the contract.

3.1 Abstracts — The Contracts

These interfaces define what can be done, not how. The concrete Mongoose implementation lives in frameworks/mongo/.

InterfaceFilePurposeMethods
IGenericRepository<T>generic-repository.abstract.tsCRUD contract for any collection. Generic type T = the document type.getAll, get, find, findOne, findWithPagination, count, create, update, updateMany, delete, findUnique, aggregate
IDataServicesdata-service.abstract.tsSingle entry point to all repositories. Inject this into use-cases.students, marks (typed repos)

3.2 Entities — Domain Models

Entities define the shape of your data in pure TypeScript — no schema, no annotations. Think of them as "what a Student looks like in our system".

// src/core/entities/student.entity.ts export interface IStudent { name: string; email: string; // unique across all students age: number; grade: string; phone?: string | null; address?: string | null; is_active: boolean; // soft-delete flag } export interface IStudentDocument extends IStudent, Document {} // adds _id, createdAt, updatedAt

5. Frameworks/Mongo Layer

Location: src/frameworks/mongo/  —  This is the only place in the entire project where Mongoose code exists. If you need to add a database query method, it goes here.

5.1 MongoGenericRepository — How Every DB Method Works

MethodMongoose Behind the ScenesWhen to Use
getAll()Model.find().populate()List all documents (no filter).
get(id)Model.findById(id).populate()Fetch one document by _id.
find(filter)Model.find(filter)Filter documents. Supports $or, $regex, etc.
findOne(filter)Model.findOne(filter)Find one document by any field (e.g., email).
findWithPagination()Model.find().skip().limit().sort()Paginated list. skip = (page-1)×limit.
count(filter)Model.countDocuments(filter)Count matching docs. Used for totalPages.
create(item)new Model(item).save()Insert a new document.
update(id, item)Model.findByIdAndUpdate(id, item, {new:true})Update and return updated doc.
updateMany(filter, update)Model.updateMany(filter, update)Bulk update multiple docs.
delete(id)Model.findByIdAndDelete(id)Hard delete a document.
findUnique(data)Model.findOne(data)Check if a document exists by unique field.
aggregate(pipeline)Model.aggregate(pipeline)Complex queries: joins, grouping, reshaping.
aggregateWithPagination()Model.aggregate([ ...pipeline, {$facet}])Paginated aggregation in ONE query using $facet.

5.2 MongoDataServices — Wiring Models to Repositories

This class implements IDataServices. It wires each Mongoose model to a MongoGenericRepository instance. Think of it as the "registry" of all collections.

// src/frameworks/mongo/mongo-data-services.service.ts class MongoDataServices implements IDataServices { students: IGenericRepository<IStudentDocument>; marks: IGenericRepository<IMarksDocument>; constructor() { this.students = new MongoGenericRepository(Student); // Student = Mongoose model this.marks = new MongoGenericRepository(Marks); // To add Course: this.courses = new MongoGenericRepository(Course); } }

5.3 Mongoose Schema — Key Options Explained

timestamps: true
Mongoose auto-adds createdAt and updatedAt to every document. You never have to set these manually.
versionKey: false
Removes the __v field that Mongoose adds by default. Cleaner API responses without this internal field.
unique: true
Creates a unique B-tree index in MongoDB. Duplicate inserts throw error code 11000, caught by error middleware.
ref: 'Student'
Tells Mongoose which model the ObjectId refers to. Required for .populate() and helps IDEs/tooling understand the relationship.

7. Controllers, Routes & Folder-Based Swagger

7.1 Controllers — Thin HTTP Layer

Controllers do exactly 3 things and nothing more: (1) Validate the request body using a DTO, (2) Call the use-case method, (3) Send the response. Business logic must never be written in a controller.
// Controller pattern — same structure for every endpoint export const createStudent = asyncHandler(async (req, res) => { // Step 1: Validate input using DTO (Joi) const { error, value } = createStudentSchema.validate(req.body, { abortEarly: false }); if (error) return sendError(res, 'Validation failed', 400, error.details.map(d => d.message)); // Step 2: Call use-case (all business logic lives here) const student = await studentUseCase.createStudent(value); // Step 3: Send response using standard envelope sendSuccess(res, 'Student created successfully', student, 201); });

Traditional approach puts all Swagger in one file → 2000+ lines for 10 resources. This project uses module-owned swagger files.

// config/swagger.ts — just spreads all module swagger specs together import { studentPaths, studentSchemas } from '../routes/student/student.swagger'; import { marksPaths, marksSchemas } from '../routes/marks/marks.swagger'; export const swaggerSpec = { paths: { ...studentPaths, ...marksPaths }, // All module paths merged here components: { schemas: { ...studentSchemas, ...marksSchemas } }, }; // Adding Course: just add ...coursePaths and ...courseSchemas — zero changes elsewhere

7.3 asyncHandler — Why Every Controller Uses It

// Without asyncHandler: if use-case throws, Express hangs the request silently router.get('/', async (req, res) => { await useCase.doSomething(); }); // ❌ WRONG // With asyncHandler: promise rejection → next(err) → error middleware → JSON error response router.get('/', asyncHandler(async (req, res) => { await useCase.doSomething(); })); // ✅ CORRECT

7.4 Standard Response Envelope

Every API response — success or error — follows the same JSON structure:

// Success { "success": true, "message": "Student created", "data": { ... } } // Error { "success": false, "message": "Validation failed", "errors": ["Name is required"] }

8. Step-by-Step: Creating a New CRUD Module

Example: Adding a Course resource. Follow these 7 steps in order. Each step maps to one folder in the architecture.
1

Create the Entity (Domain Model)

Create src/core/entities/course.entity.ts

Define a TypeScript interface ICourse with all fields (name, code, teacher_id, etc.). Also export ICourseDocument extends ICourse, Document for Mongoose typing.

Rule: No Mongoose types here. Plain TypeScript interfaces only.

2

Create the DTOs (Input Validation)

Create src/core/dtos/course/create-course.dto.ts and update-course.dto.ts

Write Joi schemas for the POST body (required fields) and PATCH body (all optional, min:1). Export both the Joi schema and the TypeScript interface.

3

Create the Mongoose Model

Create src/frameworks/mongo/model/course.model.ts

Define a new Schema<ICourseDocument>({...}) with the fields, types, required flags, and constraints. Export const Course = model('Course', courseSchema).

Add timestamps: true, versionKey: false as schema options always.

4

Register the Repository in MongoDataServices

Open src/frameworks/mongo/mongo-data-services.service.ts

Add: this.courses = new MongoGenericRepository<ICourseDocument>(Course);

Also add courses: IGenericRepository<ICourseDocument>; to src/core/abstracts/data-service.abstract.ts.

5

Create the Use-Case (Business Logic)

Create src/use-cases/course/course.use-case.ts

Write getAllCourses(), getCourseById(), createCourse(), updateCourse(), deleteCourse(). Only use this.dataServices.courses.*. Throw named errors for not-found cases.

6

Create the Controller

Create src/controllers/course.controller.ts

One exported function per endpoint. Each: validates with DTO → calls use-case → sends response. Always wrap with asyncHandler(). Use sendSuccess() and sendError().

7

Create Routes + Swagger File + Mount

Create folder src/routes/course/ with three files:

① course.routes.ts — Express Router. Map GET/POST/PATCH/DELETE to controller functions.

② course.swagger.ts — Export coursePaths and courseSchemas objects.

③ In src/routes/index.ts add: router.use('/courses', courseRoutes);

④ In src/config/swagger.ts add: ...coursePaths and ...courseSchemas.

Never skip the DTO step (Step 2). Raw req.body should never reach the database or use-case. Joi validation at the controller level is the first line of defence against bad or malicious data.

9. Step-by-Step: Adding a MongoDB $lookup (Join)

$lookup is MongoDB's equivalent of SQL LEFT JOIN. It fetches matching documents from another collection and attaches them to each document. In this project, the marks module uses $lookup to attach student info to every marks record.

How $lookup Works (Annotated)

// Inside a use-case method — build the aggregation pipeline const pipeline = [ // STEP A: Filter documents first (do this BEFORE lookup for performance) { $match: { is_active: true } }, // STEP B: $lookup — JOIN another collection { $lookup: { from: 'students', // Target collection name (MongoDB name, lowercase+plural) localField: 'student_id', // Field in THIS collection (the FK) foreignField: '_id', // Field in TARGET collection (usually the PK) as: 'student', // Name of the output array attached to each doc }, // Result: each document now has student: [{ _id, name, email, ... }] }, // STEP C: $unwind — flatten the array [ {student} ] → {student} { $unwind: { path: '$student', preserveNullAndEmptyArrays: true, // Keep doc even if no student match (LEFT JOIN) }, // Result: student is now a plain object, not an array }, // STEP D: $project — shape the final output (SELECT which fields) { $project: { _id: 1, hindi: 1, english: 1, math: 1, total: 1, percentage: 1, student: { // Only expose selected student fields _id: '$student._id', name: '$student.name', email: '$student.email', }, }, }, ]; return this.dataServices.marks.aggregate(pipeline);

Where to Write Lookup Code

1

Write the aggregation pipeline inside the Use-Case

Open src/use-cases/[module]/[module].use-case.ts and build a private method like private buildWithStudentPipeline(match?) that returns the pipeline array.

2

Call aggregate() from IDataServices

Call this.dataServices.marks.aggregate(pipeline) — this runs the full $lookup pipeline against MongoDB.

3

For paginated lookup — use $facet

Add a $facet stage at the end of the pipeline to get data + totalCount in ONE query. See getAllMarks() in marks.use-case.ts as the reference.

Common $lookup Patterns

PatternfromlocalFieldforeignFieldUse Case
Marks → Studentstudentsstudent_id_idShow marks with student name/email
Enrollment → Coursecoursescourse_id_idShow enrollment with course details
User → Rolerolesrole_id_idShow user with role name/permissions
Order → Productproductsproduct_id_idShow order with product name/price
Book → Authorauthorsauthor_id_idShow book with author bio
Collection naming: MongoDB collection names are lowercase + plural. Mongoose converts model('Student', ...) → collection name students. Always use the collection name (not the model name) in the from field of $lookup.

11. Key Concepts for Learners

🗄️
Repository Pattern
Abstracts all database operations behind an interface (IGenericRepository). Business logic never imports Mongoose. It only calls methods like find(), create() on the interface.
Rule: The only place with Model.findById() is in MongoGenericRepository.
⚙️
Use-Case Pattern
Each feature has a use-case class that contains all the business rules for that domain. Controllers call use-cases. Use-cases call repositories. No layer is skipped.
Rule: Never put if/else business logic in a controller.
📋
DTO Pattern
Data Transfer Objects define the exact shape and validation rules for incoming API data. Joi schemas enforce types, required fields, and ranges at the HTTP boundary before data enters the system.
Rule: Validate req.body before calling any use-case method.
💉
Dependency Injection
Use-cases receive IDataServices through the constructor. They don't create it themselves. This makes swapping implementations (real DB vs mock) trivial without changing the use-case code.
constructor(private readonly dataServices: IDataServices)
🔗
MongoDB $lookup (JOIN)
Fetches related documents from another collection and attaches them. Used in the marks module to embed student info into every marks response without making two separate API calls from the client.
$lookup → $unwind → $project in the aggregation pipeline.
📊
$facet Pagination
$facet runs multiple sub-pipelines in parallel on the same dataset. Used to get paginated data AND total count in a single MongoDB query — twice as efficient as two separate queries.
$facet: { data: [$skip,$limit], totalCount: [$count] }
🛡️
Soft Delete
Instead of removing a document from the database permanently, set is_active: false. The data stays for audit/history purposes. Filter is_active: true on list queries to hide inactive records.
update(id, { is_active: false }) instead of delete(id)
🌐
Microservice Readiness
Each resource module (student/, marks/) is self-contained with its own routes, swagger, use-case, and DTOs. Extracting any one of them into a separate microservice only requires moving the folder + wiring a new data source.
Each module folder = potential microservice boundary.

12. Quick Cheat Sheet — Files to Create for a New API

When adding a new resource (example: Teacher), create these files in exactly this order. Each file corresponds to one architecture layer.
#File to CreateLayerWhat to WriteTime Est.
1 core/entities/teacher.entity.ts Core ITeacher interface + ITeacherDocument extends ITeacher, Document 5 min
2 core/dtos/teacher/create-teacher.dto.ts Core Joi object with required fields. Export interface + schema. 5 min
3 core/dtos/teacher/update-teacher.dto.ts Core Same fields but all .optional(). Add .min(1) on the object. 3 min
4 frameworks/mongo/model/teacher.model.ts Framework new Schema<ITeacherDocument>. Add timestamps:true, versionKey:false. Export model. 8 min
5 core/abstracts/data-service.abstract.ts Core ADD line: teachers: IGenericRepository<ITeacherDocument>; 1 min
6 frameworks/mongo/mongo-data-services.service.ts Framework ADD in constructor: this.teachers = new MongoGenericRepository(Teacher); 2 min
7 use-cases/teacher/teacher.use-case.ts Use-Cases Class with getAll, getById, create, update, delete methods. Use this.dataServices.teachers only. 15 min
8 controllers/teacher.controller.ts Controllers One export per endpoint. Validate DTO → call use-case → sendSuccess. Wrap all in asyncHandler. 10 min
9 routes/teacher/teacher.routes.ts Routes Express Router. Map GET/POST/PATCH/DELETE to controller functions. 5 min
10 routes/teacher/teacher.swagger.ts Routes/Swagger Export teacherPaths (API endpoints) and teacherSchemas (request/response shapes). 10 min
11 routes/index.ts Routes ADD: router.use('/teachers', teacherRoutes); 1 min
12 config/swagger.ts Config ADD: ...teacherPaths and ...teacherSchemas in the spread. 2 min
Total estimate: ~67 minutes for a full CRUD module with Swagger documentation. Files 1–3 never touch the database. Files 4–6 are pure copy-paste-adapt from existing models. Files 7–12 contain the actual unique logic for your resource.
